## 第一章：产品愿景与定位

### 1.1 一句话定义

**OpenAGI** 是一款面向中国市场的个人 AI 超级助手桌面应用。

### 1.2 产品愿景

OpenAGI 是一款个人 AI 助手桌面应用，具备：
- 🎨 从 ClawX（桌面应用界面框架）继承 100% 的 UI/UX（用户界面和体验）
- 🧠 集成 OpenClaw（开源 AI 网关）的多渠道 AI 编排引擎，继承 100% 的功能
- ⚡ 利用 Claude 的 AI Code（代码交互）核心逻辑，继承 100% 的优点功能
- 🚀 提供个性化扩展和定制的基础，最少的删减三个软件原来的代码和功能，最终做出一个超级新软件

### 1.3 核心定位

```
OpenAGI = ClawX 的桌面 GUI 壳
        + OpenClaw 的多渠道 AI 内核
        + Claude Code 的 AI 编程交互引擎
```

一句话：一个能在 WhatsApp/Telegram/Slack/飞书/钉钉/QQ/微信 上帮你干活的 AI 桌面助手，同时具备像 Claude Code 一样的代码编写和自主执行能力。

### 1.4 未来规划

| 方向 | 说明 |
|------|------|
| 📱 手机端 | 未来会推出 iOS 和 Android 手机版 |
| 🔒 端到端加密 | 所有通信采用端到端加密，保护隐私 |
| 🔀 独立发展 | 与 OpenClaw 不再同步更新，走自己的路线 |
| 🔗 技能共享 | 但仍可以共用 OpenClaw 的 Skill（技能）生态 |
| 🇨🇳 中文生态圈 | 以中文为主的插件、技能、社区生态 |

### 3.2 Claude Code 引擎集成方案：Rust FFI（外部函数接口）

**通俗解释**：
- FFI（Foreign Function Interface，外部函数接口）就像"翻译接口"
- 我们把 Claude Code 的 Rust 代码编译成一个"动态库"（可以理解为一个功能包）
- 然后 OpenAGI 的 Electron 主进程通过 koffi/napi（两种调用工具）直接调用这个功能包
- 这样做性能最好，就像直接面对面说话，不需要中间人转达

**架构详细设计**：

```
OpenAGI Electron 主进程
    │
    ├── OpenClaw Gateway 管理器
    │   │
    │   └── OpenClaw 子进程（WebSocket 端口 18799）
    │       ├── 多渠道消息收发
    │       ├── 智能体调度
    │       └── 技能执行
    │
    └── Claw Code Engine 管理器
        │
        ├── libclaw.dylib（macOS 动态库）
        │   或 claw.dll（Windows 动态库）
        │   或 libclaw.so（Linux 动态库）
        │
        └── 通过 koffi/napi-rs 调用：
            ├── claw_session_create()    → 创建新会话
            ├── claw_session_send()      → 发送消息并获取回复
            ├── claw_session_stream()    → 流式获取回复
            ├── claw_tool_execute()      → 执行工具
            ├── claw_session_resume()    → 恢复历史会话
            ├── claw_session_compact()   → 压缩会话历史
            └── claw_session_destroy()   → 销毁会话
```

**备选方案**：如果 FFI 编译遇到困难，降级为子进程方案（通过 stdin/stdout 通信），两种方案的接口设计保持一致，可以随时切换。

### 3.3 统一路由（智能意图识别）

**通俗解释**：用户不需要手动切换"聊天模式"和"编程模式"，OpenAGI 会自动判断你想做什么。

```
用户输入
    │
    ▼
┌──────────────────────────┐
│    意图识别路由器          │
│    （自动判断用户想做啥）  │
└──────────┬───────────────┘
           │
     ┌─────┴─────┐
     │           │
     ▼           ▼
┌─────────┐ ┌─────────┐
│ 普通对话 │ │ 代码任务 │
│ 渠道消息 │ │ 文件操作 │
│ 技能调用 │ │ 系统命令 │
└────┬────┘ └────┬────┘
     │           │
     ▼           ▼
  OpenClaw    Claude Code
  Gateway      Engine
```

**判断规则示例**：
- "帮我写一个 Python 脚本" → Claude Code Engine
- "把这条消息转发到微信群" → OpenClaw Gateway
- "总结一下今天的聊天记录" → 两个引擎协作
- "帮我修改 config.json 文件" → Claude Code Engine
- "给张三发一条飞书消息" → OpenClaw Gateway

## 第九章：风险评估与应对

| 风险 | 影响 | 应对方案 |
|------|------|---------|
| Rust FFI 编译失败 | 无法集成代码引擎 | 降级为子进程方案 |
| 跨平台编译困难 | 只能在当前系统运行 | P0 仅支持 macOS arm64 |
| 插件体积过大 | 安装包臃肿 | 只打包核心插件，其余按需下载 |
| 国产大模型 API 差异 | 接入成本高 | 优先接入 OpenAI 兼容格式的（如 DeepSeek） |
| 微信集成限制 | 微信官方限制第三方 | 使用成熟方案（如 itchat 或 wechaty） |
| 性能问题 | 双引擎可能占用资源多 | 按需启动引擎，不用时休眠 |
