## 一、计划理念差异说明

| 维度 | 计划A（原版） | 计划B（本版） |
|------|-------------|-------------|
| 策略 | 线性串行（验证→修复→开发→交付） | 双轨并行（质量线 + 功能线同步推进） |
| 排期逻辑 | 按阶段依赖顺序 | 按业务价值和风险等级排优先级 |
| 交付节奏 | 4个大阶段，最后才可见 | 每周一个可演示里程碑（Demo Friday） |
| 风险策略 | 发现后修复 | 前置技术探针（Spike），72小时内验证高风险项 |
| 团队模式 | 全员同阶段 | 拆为两条独立价值流，减少互相阻塞 |

## 四、推进计划（5个里程碑）

### 里程碑 0：技术探针（Day 1-3，72小时）

> **目的**：用最小成本验证最大风险项，为后续计划提供数据支撑。

| 编号 | 探针任务 | 负责人 | 成功标准 | 失败预案 |
|------|---------|--------|---------|---------|
| S1 | `pnpm install && pnpm run typecheck` | 后端 | 零类型错误或错误 < 20 | 评估重写成本，上报高级项目经理 |
| S2 | `pnpm run lint && pnpm test` | 质检 | lint 错误 < 50，测试通过率 > 80% | 制定修复优先级清单 |
| S3 | Docker 沙盒本机可启动 | 后端 | 容器正常运行并响应健康检查 | 切换为子进程方案（Plan B1） |
| S4 | Trinity 三AI协作一次完整循环 | 后端 | AI-1提案→AI-2审计→AI-3决策 跑通 | 简化为单AI+规则引擎（Plan B2） |
| S5 | 密钥生成 + 身份绑定端到端 | 后端 | 本地生成密钥并派生地址 | 接入现有加密库替代自研 |

## 七、与计划A的关键区别总结

1. **前置探针**：不盲目启动修复，先用3天验证假设，用数据说话
2. **双轨并行**：设计和开发不互相等待，Alpha 稳定基础的同时 Beta 产出原型
3. **每周可见**：Demo Friday 让用户每周看到进展，不是等到最后才交付
4. **降级预案**：每个高风险项都有 Plan B1/B2，不被单点失败卡死
5. **接口前置**：M1 阶段就定义接口协议，避免后期联调反复
