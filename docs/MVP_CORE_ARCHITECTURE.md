# OpenAGI MVP 核心架构图

> 版本：v0.1.0 | 日期：2026-04-14 | 状态：正式版

---

## 一、设计原则

| 原则 | 说明 |
|------|------|
| **生命化架构** | 沿用 openagi_m2 的生物学隐喻（cortex/heart/ghost/soul/flesh），不是工具而是数字生命 |
| **Trinity 治理** | 三 AI 博弈架构（扩张者-审计者-治理者），每个决策经三方辩论，源自 NewClaw v6 |
| **记忆先行** | 永久记忆 + 分层唤醒是 MVP 核心能力，不是附属功能 |
| **永生设计** | 从 Day 1 就设计为分布式常驻，而非事后改造 |
| **插件经济** | 经济自主能力（钱包/收付）作为未来插件，MVP 不做 |
| **资产复用** | 优先复用 openagi_m2 / NewClaw v6 / OpenClaw 已验证代码 |

## 四、永远记忆系统详细设计（MVP 核心能力）

对标 [MemPalace](https://github.com/MemPalace/mempalace) 设计 + 复用 OpenClaw memory-core（29,743行生产级代码）。

### 核心理念

**永远记忆 = 永不遗忘 + 智能蒸馏 + 分层唤醒**

传统AI每次对话从零开始。OpenAGI 的记忆是**永久的**——所有经验都持久化存储，通过三阶段梦境蒸馏自动提炼精华，通过分层唤醒机制按需激活相关记忆。

### 四层记忆架构

| 层级 | 名称 | 存储 | 容量 | 衰减机制 | 唤醒条件 |
|------|------|------|------|---------|---------|
| L0 | 热记忆 (Working) | 内存 | 当前对话上下文 | 对话结束即转温 | 始终激活 |
| L1 | 温记忆 (Recent) | SQLite向量库 | 最近 N 天交互 | 指数衰退（半衰期30天） | 语义相似度 > 0.75 |
| L2 | 冷记忆 (Archive) | SQLite + Quarto MD | 全部历史（永不删除） | 不衰减，检索需触发 | 显式查询 / 蒸馏引用 |
| L3 | 核心记忆 (Core/DNA) | JSON + Git | 身份、价值观、关键学习 | **永不衰减** | 始终注入 system prompt |

### 短期提升算法（Short-Term Promotion）

决定哪些温记忆值得被提升（巩固）到长期存储：

```
综合提升分 = 频率权重(0.24) × 访问频率
           + 相关性权重(0.30) × 搜索相关性
           + 多样性权重(0.15) × 结果多样性
           + 最近性权重(0.15) × 最近访问度
           + 巩固权重(0.10) × 蒸馏阶段信号
           + 概念权重(0.06) × 概念关联度

提升条件（同时满足）:
  · 综合分 ≥ 0.75
  · 被回忆次数 ≥ 3
  · 被不同查询命中 ≥ 2
```

### 混合检索系统

```
用户查询 / AI 内部检索
    │
    ├──▶ 向量搜索（语义相似度）
    │      使用嵌入模型生成查询向量
    │      余弦相似度匹配
    │
    ├──▶ 关键字搜索（BM25全文检索）
    │      支持 CJK 中文分词
    │      精确匹配 + 模糊匹配
    │
    └──▶ 合并排序
           │
           ├─ 混合分 = α × 向量分 + (1-α) × 关键字分
           │
           ├─ MMR 去重（λ=0.7 相关性, 0.3 多样性）
           │   score(i) = λ × relevance(i) - (1-λ) × max_sim_to_selected
           │
           └─ 时间衰退加权
               decayed = score × e^(-λ × age_days)
               半衰期 = 30天
```

### 关键参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 短期回忆窗口 | 16天 | 温记忆最大回溯范围 |
| 提升最小分数 | 0.75 | 从温→冷的最低阈值 |
| 提升最小回忆次数 | 3次 | 至少被回忆3次才考虑提升 |
| 提升最小唯一查询 | 2个 | 至少被2种不同场景命中 |
| 概念标签上限 | 8个 | 每条记忆最多8个概念标签 |
| MMR Lambda | 0.7 | 相关性70%，多样性30% |
| 时间衰退半衰期 | 30天 | 温记忆的默认衰退周期 |
| Light Sleep 周期 | 2小时 | 轻度蒸馏频率 |
| REM Sleep 周期 | 6小时 | 中度蒸馏频率 |
| Deep Dreaming 周期 | 24小时 | 深度蒸馏频率 |
| 蒸馏最近性半衰期 | 14天 | 蒸馏信号的权重衰退 |

## 六、可配置多核治理引擎详细设计

### 核心理念

**不是固定3核或4核，而是用户可配置1-4核，每个核可选模型。**
权限系统可自动升核——用户用1核省钱，但遇到高风险操作时系统自动临时升核。

### 四种模式的流程

```
┌─ 1核模式（直接问答）────────────────────────────┐
│                                                 │
│  用户请求 → Core-0 直接回答 → 输出               │
│                                                 │
│  适用：闲聊、简单查询、低风险操作                  │
│  成本：1次LLM调用                                │
└─────────────────────────────────────────────────┘

┌─ 2核模式（单审计）──────────────────────────────┐
│                                                 │
│  用户请求 → Core-0 生成方案                      │
│                  │                              │
│                  ▼                              │
│             Core-1 审计                          │
│                  │                              │
│             通过 → 执行 / 阻止 → 反馈用户         │
│                                                 │
│  适用：一般任务、代码编写、文档生成                 │
│  成本：2次LLM调用（串行）                         │
└─────────────────────────────────────────────────┘

┌─ 3核模式（双审计并行）──────────────────────────┐
│                                                 │
│  用户请求 → Core-0 生成方案                      │
│                  │                              │
│          ┌───────┴───────┐                      │
│          ▼               ▼                      │
│     Core-1 审计     Core-2 审计                  │
│          │               │                      │
│          └───────┬───────┘                      │
│                  ▼                              │
│          Core-0 综合定稿                         │
│                                                 │
│  适用：重要决策、复杂任务                          │
│  成本：3次LLM调用（1串行 + 2并行）                │
└─────────────────────────────────────────────────┘

┌─ 4核模式（三审计并行）──────────────────────────┐
│                                                 │
│  用户请求 → Core-0 生成方案                      │
│                  │                              │
│       ┌──────────┼──────────┐                   │
│       ▼          ▼          ▼                   │
│  Core-1审计  Core-2审计  Core-3审计              │
│       │          │          │                   │
│       └──────────┼──────────┘                   │
│                  ▼                              │
│       Core-0 综合定稿（含贡献分）                 │
│                                                 │
│  适用：关键操作、高风险、财务相关                   │
│  成本：4次LLM调用（1串行 + 3并行）                │
└─────────────────────────────────────────────────┘
```

### 权限自动升核矩阵

| 权限级别 | 操作类型 | 最低核数 | 用户选1核时 | 示例 |
|---------|---------|---------|-----------|------|
| L0 | 纯内部操作 | 1核 | 正常执行 | 写日志、生成草稿 |
| L1 | 受限外部读取 | 2核 | 自动升到2核 | 查询API、沙箱测试 |
| L2 | 低风险真实操作 | 3核 | 自动升到3核 | 测试网转账 |
| L3 | 高风险真实操作 | 4核+人机双签 | 自动升到4核 | 真实钱包操作 |
| L4 | 永久禁止 | — | 直接拒绝 | 绕过约束、删除账本 |

> 升核是临时的——执行完毕后自动恢复用户设定的核数。

## 十二、巡检AI（Commander Core）

### 定位

巡检AI不是第5个治理核心，而是**坐在"用户位置"上的AI指挥官**——它代替用户发出指令，但指令仍然经过多核治理流程审查。

```
                    用户 / 巡检AI（Commander）
                    ┌──────────────────────┐
                    │  用户手动输入          │
                    │       或              │
                    │  巡检AI自动生成指令    │◄── 定时 + 事件触发
                    └──────────┬───────────┘
                               │
                               ▼
                    ┌──────────────────────┐
                    │  可配置多核治理引擎    │
                    │  1-4核 + 权限自动升核  │
                    └──────────────────────┘
```

### 触发机制（双模式）

#### 模式A：定时触发

用户可设定频率，范围 **1分钟 ~ 1年**，完全自定义：

| 预设 | 适用场景 |
|------|---------|
| 1分钟 | 高频监控（如实时交易） |
| 5分钟 | 密集开发期 |
| 10分钟 | 日常工作（默认） |
| 30分钟 | 轻度监控 |
| 1小时 | 后台运行 |
| 自定义 | 输入任意间隔（1分钟 ~ 1年） |

#### 模式B：事件触发（智能等待）

**核心原则：不打断正在执行的任务，等任务完成后再巡检。**

```
定时器到了 → 检查当前是否有任务在执行
                │
          ┌─────┴─────┐
          ▼           ▼
      无任务在执行   有任务在执行（如编程30分钟）
          │           │
          ▼           ▼
      立即巡检     注册"任务完成"事件监听
                      │
                      ▼
                  任务完成时触发巡检
                      │
                      ▼
                  巡检内容包含刚完成的任务结果
```

**支持的事件触发器：**

| 事件 | 说明 | 巡检动作 |
|------|------|---------|
| `task:completed` | 任何任务执行完毕 | 总结任务结果，规划下一步 |
| `task:failed` | 任务失败 | 分析失败原因，决定重试或放弃 |
| `task:blocked` | 任务被审计阻止 | 评估是否调整方案 |
| `entropy:crisis` | 熵值进入危机 | 紧急评估，决定是否降级 |
| `memory:distill` | 知识蒸馏完成 | 检查新知识，更新计划 |
| `schedule:timer` | 定时器到期 | 常规巡检 |
| `user:idle` | 用户长时间无输入 | 自主规划下一步 |

#### 触发优先级规则

```
1. 事件触发 > 定时触发（事件发生时重置定时器）
2. 任务执行中 → 定时巡检延迟到任务完成
3. 危机事件 → 无论如何立即触发（不等待）
4. 多个事件同时到达 → 合并为一次巡检
```

### 巡检工作循环

```
┌─ Step 1: 信息收集 ──────────────────────────┐
│  · 读取记忆系统（最近的任务和进展）            │
│  · 检查心绪状态（熵值、效价）                 │
│  · 查看未完成任务队列                         │
│  · 检查触发事件的上下文                       │
└──────────────────┬──────────────────────────┘
                   ▼
┌─ Step 2: 总结 + 规划 ──────────────────────┐
│  · 生成近期进展摘要                          │
│  · 识别阻塞项和风险                          │
│  · 规划下一步最优行动                         │
│  · 生成一条最佳指令                          │
└──────────────────┬──────────────────────────┘
                   ▼

│  草稿模式：填入发送框，用户可修改/发送/取消    │

└─────────────────────────────────────────────┘
```

### 两种发送模式

| 模式 | 行为 | 安全限制 | 适用场景 |
|------|------|---------|---------|
| **草稿模式**（默认） | 指令填入发送框，等用户确认 | 无限制 | 需要人类监督 |
| **自动执行模式** | 直接发送并触发治理流程 | 最高只执行L1操作，L2+仍需人类确认 | 陛下不在时自主运行 |

### 核心概念

### 深度聊天 vs AI团队群聊

| 维度 | 深度聊天 | AI团队群聊 |
|------|---------|-----------|
| **参与者** | 用户 ↔ 1个AI（Core-0）+ 可选审计核心 | 用户 + N个AI成员（各有独立技能） |
| **对话方式** | 一对一深度对话 | 群聊，成员间可互相@交流协作 |
| **治理** | 1-4核治理引擎 | 团队内部自行协作和审核 |
| **模式** | 单一对话模式 | **讨论模式** ↔ **工作模式** 无缝切换 |
| **适用场景** | 深度思考、单任务、个人助理 | 复杂项目、多角色协作、并行开发 |
| **成本** | 低（1-4次LLM） | 按参与成员数计（可控：只@需要的人） |

### AI团队协作机制

对标 [openteams](https://github.com/openteams-lab/openteams) 的核心设计，实现以下六大能力：

#### 能力1：上下文共享

所有AI成员在同一对话空间工作，自动共享：
- 项目背景和讨论历史
- 代码变更和文件状态
- 其他成员的发言（即使未被@，也能"旁听"积累上下文）

```
群聊上下文 = 共享记忆
    │
    ├─ 所有成员的发言历史
    ├─ 用户上传的文件和代码
    ├─ 项目当前状态（从记忆系统注入）
    └─ 每位成员"旁听"后的内部摘要
```

#### 能力2：@沟通机制

```
用户消息: "讨论登录方案 @界面设计师 @前端开发者"
    │
    ▼
系统解析被@的成员
    │
    ├─ 界面设计师（被@）→ 生成回复
    ├─ 前端开发者（被@）→ 生成回复
    └─ 安全工程师（未被@）→ 旁听，积累上下文
    
AI回复: "@前端开发者 这个方案可行吗？"
    │
    ▼
系统检测到AI之间的@
    │
    └─ 前端开发者 → 自动被触发，形成协作链条
```

| 规则 | 说明 |
|------|------|
| **用户@指定成员** | 只有被@的成员回复（节省成本） |
| **用户@全体** | 所有在线成员都回复 |
| **AI互相@** | 代理之间互相@触发任务交接，自动形成协作链条 |
| **轮次上限** | AI间链式对话最多N轮（用户可设，默认3轮），防止无限循环 |
| **旁听者** | 未被@的成员默默旁听，下次被@时知道全部前因后果 |
| **用户随时插入** | 用户可以随时发消息打断或重定向AI讨论 |

```
┌─ 讨论模式 (Discussion) ──────────────────────────┐
│                                                   │

│  特点：轻量、快速、多视角                           │
│                                                   │
│  [切换到工作模式 ▶]                                │
└───────────────────────────────────────────────────┘

┌─ 工作模式 (Work) ────────────────────────────────┐
│                                                   │
│  成员各自领取任务并行执行                            │
│  输出：代码、文件、测试结果、部署                    │

│                                                   │
│  进度面板：                                        │
│  ✅ 界面设计师 — 登录页Figma完成                    │

│                                                   │

```

#### 能力4：并行处理与任务自动投递

```

    ▼

    │
```

## 十五、技能系统（Skill Engine）

### 核心概念

技能是AI成员的**核心能力体现**。通过为AI成员配置不同的技能，赋予他们不同的工作能力和职责。

> **技能 = 封装了一类具体能力的模块**，例如代码生成、信息检索、数据处理、文件操作或调用外部API。
> 每个技能包含明确的输入输出定义、执行逻辑以及与环境交互的方式。
> 多个技能可以组合使用，形成更复杂的工作流。

### 技能的组成

```python

class Skill:
    id: str                    # 唯一标识 (如 "code-gen")
    name: str                  # 中文名 (如 "代码生成")
    name_en: str               # 英文名 (如 "Code Generation")
    description: str           # 技能描述
    category: str              # 分类 (开发/设计/数据/通信/文件/API)
    
    # 能力定义
    inputs: list[ParamDef]     # 输入参数定义
    outputs: list[ParamDef]    # 输出参数定义
    execute: Callable          # 执行逻辑
    
    # 配置
    required_permissions: str  # 所需权限级别 (L0-L3)
    auto_downloadable: bool    # 是否允许AI自动下载安装
    compatible_agents: list    # 兼容的Agent代理类型
```

## 十六、免API浏览器搜索引擎（Browser Engine）

### 核心理念

**你的浏览器就是 API。** 不需要任何搜索 API Key，利用用户已有的浏览器登录状态和 Cookies，直接在真实 Chrome 浏览器中执行搜索和数据抓取。

对标 [bb-browser](https://github.com/epiral/bb-browser) 完整实现。

### 三层架构

```
┌─────────────────────────────────────────────────────────┐
│  AI 层（OpenAGI 核心引擎 / AI 团队成员）                  │
│  · AI 需要搜索信息时，调用浏览器技能                       │
│  · 通过 MCP 协议或内部 API 发送命令                       │
└────────────────────────┬────────────────────────────────┘
                         │ HTTP / MCP
┌────────────────────────▼────────────────────────────────┐
│  中间层（本地守护进程）                                    │
│  · 接收 AI 的搜索/抓取请求                                │
│  · 转换为浏览器操作命令                                    │
│  · 管理标签页生命周期                                      │
│  · 返回结构化结果                                         │
└────────────────────────┬────────────────────────────────┘
                         │ CDP (Chrome DevTools Protocol)
┌────────────────────────▼────────────────────────────────┐
│  浏览器层（真实 Chrome）                                   │
│  · 在标签页内执行 JavaScript (eval)                       │
│  · fetch() 自动携带用户已有的 Cookies                     │
│  · 请求看起来完全来自真实用户                               │
│  · 无需 API Key，无速率限制，免费                          │
└─────────────────────────────────────────────────────────┘
```

### 为什么不需要 API

| 传统方案 | bb-browser 方案 |
|---------|----------------|
| 申请 Google Search API Key | 直接在 Chrome 中打开 Google 搜索 |
| 付费 $5/1000次查询 | 免费，无限次 |
| API 返回结构化但有限的结果 | 获取页面完整内容 |
| 需要每个平台单独接入 API | 所有已登录的网站都可以直接访问 |
| 被封号/限流风险 | 正常用户行为，不触发反爬 |

```
                     （群聊结论作为Core-0的输入）
                              │
                              ▼
                     审计核心审查群聊方案
                              │
                              ▼
                     执行 / 拒绝
```

群聊和多核治理是**两个不同层面**的系统：
- **群聊** = 多AI协作讨论，产出方案（创造阶段）
- **多核治理** = 方案的安全审查和批准（治理阶段）

群聊的讨论结果需要经过治理引擎审批后才能执行。

#### 借鉴1：闭合学习循环（Learning Loop）— 最重要

```

当前 OpenAGI 的学习路径：

                           │
```

**具体实现**：当 AI 重复执行某类任务超过 N 次时，自动将操作模式提取为可复用技能，保存到 `~/.openagi/skills/`。每次使用技能后，根据结果反馈自动微调技能参数。

#### 借鉴6：子代理DAG并行（增强群聊系统）

Claude Code 的子代理支持**依赖图(DAG)**自动调度：

```
任务: "重构支付模块"
    │
    ▼
自动分解为DAG:
    
    code-gen ─────┬──→ test（依赖code-gen）
                  ├──→ doc（依赖code-gen）    ← 可并行
                  ├──→ refactor（依赖code-gen）
                  │
    test ─────────┤
    refactor ─────┼──→ review（依赖test+refactor）
                  │
    review ───────┴──→ 完成

时间线:
  T=0  code-gen 启动
  T=3  code-gen完成 → test/doc/refactor 并行启动
  T=5  doc完成
  T=7  test+refactor完成 → review启动
  T=9  review完成
  
总耗时 ~9s（串行需 30s+）
```

**整合到群聊系统**：工作模式下，AI成员的任务自动按DAG依赖排序并行执行。
