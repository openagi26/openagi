## 一、Layer 2：Agent 核心引擎

### 1. 多核治理引擎

- [ ] `[AUTO]` **1核模式**：发一条消息，Core-0 直接返回，日志中无 Core-1/2/3 启动记录，框架处理延迟 < 2 秒（不含 LLM 响应时间）
- [ ] `[MANUAL]` **2核模式**：Core-0 生成回答 → Core-1 审计 → 最终结论，UI 中 Core-0 和 Core-1 的原始输出分别显示在独立面板，各含角色标签和时间戳
- [ ] `[AUTO]` **3核并行验证**：Core-1 和 Core-2 的启动时间戳差值 < 200ms（日志可测），两者输出不互相等待
- [ ] `[MANUAL]` **4核模式**：Core-0~3 同时工作，UI 可见4个核的独立输出，综合定稿有明确来源标注
- [ ] `[AUTO]` **审计冲突仲裁**：Core-1 与 Core-2 结论矛盾时，系统按预设仲裁策略（如多数投票/Core-0优先）输出定稿，日志记录冲突详情
- [ ] `[AUTO]` **权限自动升核**：发起 L2 操作时，系统自动切换到 ≥3 核，日志记录升核原因和时间；L3 操作强制4核且等待人工确认信号
- [ ] `[AUTO]` **权限降核**：L2 操作完成后，系统在 30 秒内自动降回用户原来设定的核数，日志可验证
- [ ] `[AUTO]` **预设方案切换**：点击"[省钱]"预设，API 请求中 model 字段变为 haiku 系列，Core 数量为1，可通过拦截请求验证

### 2. 心绪引擎 (HeartEngine)

- [ ] `[AUTO]` **状态读取**：`GET /api/v1/heart/status` 返回 `{"entropy": float, "valence": float, "state": str}`，两值均 ∈ [0.0, 1.0]
- [ ] `[AUTO]` **事件响应（成功）**：触发 `llm_call_success` 事件后，entropy 减少量 ∈ [0.015, 0.025]（允许误差）
- [ ] `[AUTO]` **事件响应（失败）**：触发 `task_failed` 事件后，entropy 增加量 ∈ [0.07, 0.09]
- [ ] `[AUTO]` **边界钳位**：连续触发10次 `task_failed`，entropy 仍 ≤ 1.0，不溢出；连续触发10次 `llm_call_success`，entropy 仍 ≥ 0.0
- [ ] `[AUTO]` **状态跃迁**：entropy ≥ 0.80 时，`state` 字段变为 `"crisis"`；UI 心绪指示灯颜色变为红色
- [ ] `[AUTO]` **温度联动**：`calm` 状态下的 LLM temperature 值 < `crisis` 状态下的值，可通过 API 响应 header 或日志验证
- [ ] `[AUTO]` **持久化**：服务重启后，`GET /api/v1/heart/status` 返回的 entropy 与重启前误差 < 0.01（从 JSON 文件恢复）

### 3. ReAct+Plan 执行循环

- [ ] `[AUTO]` **Plan 生成**：给定多步任务，系统返回 Plan，格式为 JSON 数组，每项含 `step_id`、`action`、`expected_observation` 字段
- [ ] `[AUTO]` **步骤日志**：每个步骤的感知/思考/行动/观察均有对应日志行，格式为 `[REACT][step_N][phase] message`
- [ ] `[AUTO]` **失败恢复**：中间步骤返回错误时，系统生成新的 Plan 继续执行，而非抛出异常中止
- [ ] `[AUTO]` **最大步骤限制**：超过 20 步（可配置）后，系统强制终止并返回 `{"status": "max_steps_exceeded", "steps_taken": N}`，不陷入无限循环

### 4. 上下文压缩引擎

- [ ] `[AUTO]` **自动触发**：对话 token 用量超过上限的 70% 时，日志出现压缩触发记录，可通过 token 计数 API 验证
- [ ] `[AUTO]` **压缩率**：压缩后字符数 ≤ 原始字符数的 40%（即减少 ≥ 60%），可通过压缩前后字符数对比验证
- [ ] `[MANUAL]` **关键内容保留**：压缩后摘要中，原文中所有 ``` 代码块至少保留一个 hash 可匹配的版本；明显的错误信息关键词（如 `Error:`、`Exception`）至少保留一处
- [ ] `[AUTO]` **写入温记忆**：压缩完成后，L1 数据库中可查询到对应对话的压缩摘要条目（通过会话 ID 关联）
- [ ] `[AUTO]` **压缩中并发处理**：压缩进行中用户发送新消息，新消息进入队列，压缩完成后正常处理，不丢失也不报错

## 二、Layer 3：永远记忆系统

### 1. 四层记忆读写

- [ ] `[AUTO]` **L0 读写**：写入10条消息到 L0，立即读取返回相同10条，顺序一致
- [ ] `[AUTO]` **L0 → L1 转存**：对话结束（调用 `end_session` API）后，L1 中可查询到该会话的摘要条目
- [ ] `[AUTO]` **L1 语义召回**：写入"我们用 Python 开发后端"，用"项目的编程语言是什么"查询，召回结果相似度 ≥ 0.75
- [ ] `[AUTO]` **L1 语义隔离**：用"今天天气如何"查询，不应召回任何技术记忆（相似度 < 0.5）
- [ ] `[AUTO]` **L1 时间衰退**：同语义内容，30天前写入的记忆检索得分 ≤ 新记忆得分的 60%（数值可通过 API 的 score 字段验证）
- [ ] `[AUTO]` **L1 写入失败降级**：向量库不可用时，系统将内容写入本地临时文件，待恢复后自动同步，不丢失数据
- [ ] `[AUTO]` **L2 永久存储**：L1 蒸馏后，原记忆在 L2 数据库中存在完整原文；调用 DELETE API 返回 HTTP 403，且审计日志记录该尝试
- [ ] `[AUTO]` **L3 DNA 注入**：写入 DNA 条目后，下一次对话的 system prompt（通过 `/api/v1/session/start` 可见）中包含该条目原文

### 2. 三阶段梦境蒸馏

- [ ] `[AUTO]` **Phase 1 触发**：手动调用 `POST /api/v1/memory/distill/light`，返回处理条目数和标签列表，每条记忆标签数 ≤ 8，无重复标签
- [ ] `[AUTO]` **Phase 1 幂等性**：连续触发两次 Phase 1，第二次的标签集合与第一次完全相同（无重复新增）
- [ ] `[AUTO]` **Phase 2 触发**：调用 `POST /api/v1/memory/distill/rem`，返回连接报告，格式为 `[{"source_id": str, "target_id": str, "relation_type": str, "confidence": float}]`，confidence ≥ 0.6
- [ ] `[MANUAL]` **Phase 3 触发**：调用 `POST /api/v1/memory/distill/deep`，生成梦境日记文本，包含"关键发现"、"新建连接"、"矛盾检测"三个章节
- [ ] `[AUTO]` **定时调度**：Phase 1 调度日志每 2 小时出现一次触发记录，误差 ≤ 5 分钟

### 3. 混合检索系统

- [ ] `[AUTO]` **语义检索**：输入"项目的编程语言"，召回记录中包含"Python"相关条目（无关键词重叠），验证 top-1 结果
- [ ] `[AUTO]` **BM25 精确检索**：输入"openagi415"，召回结果必须包含该精确字符串的记录
- [ ] `[AUTO]` **MMR 去重**：返回5条结果时，任意两条之间的语义相似度 < 0.9（可通过 embedding 余弦计算验证）
- [ ] `[AUTO]` **混合排序权重**：排序得分 = 0.7 × 语义分 + 0.3 × 时间衰减分，top-1 结果可通过手动构造高语义+新时间的记录来验证排序正确性
- [ ] `[AUTO]` **空结果处理**：查询完全不相关内容时，返回 `{"results": [], "count": 0}`，不报错，不抛出异常

## 三、聊天系统

### 1. 深度聊天

- [ ] `[AUTO]` **流式回复**：发送消息后，前端 EventSource/WebSocket 开始收到 token 流，首个 token 到达时间 < 3 秒（不含 LLM 首token时间）
- [ ] `[MANUAL]` **Markdown 渲染**：回复中的 `**粗体**`、`# 标题`、` ``` 代码块` 正确渲染，代码块有语法高亮和复制按钮
- [ ] `[AUTO]` **计时器**：流式输出进行中，UI 每秒更新"AI正在工作...已用时N秒"，N 单调递增
- [ ] `[AUTO]` **停止按钮**：点击停止后 500ms 内，WebSocket 不再接收新 token，后端取消推理任务
- [ ] `[AUTO]` **Token 标签**：每条 AI 回复消息旁显示 token 数，数值与后端返回的 `usage.completion_tokens` 一致
- [ ] `[AUTO]` **历史持久化**：刷新页面后，当前会话的历史消息重新加载，顺序与原始顺序一致
- [ ] `[AUTO]` **断线提示**：流式输出中途网络断开，前端在 5 秒内显示"连接中断"提示，不静默失败

### 2. AI 团队群聊

- [ ] `[AUTO]` **单点 @**：消息中 `@成员A`，只有成员A产生回复，其他成员无回复记录（日志可验证）
- [ ] `[AUTO]` **@全体**：消息中 `@全体`，所有已加入群聊的成员按加入顺序依次回复，顺序在日志中可验证
- [ ] `[AUTO]` **AI 互@ 防循环**：成员A回复中含 `@成员B`，B 回复，B 的回复不再触发新的 @（即3次交互后停止）。3轮定义：用户发1条 → A回复(轮1) → B回复(轮2) → 停止，不产生轮3
- [ ] `[MANUAL]` **讨论↔工作模式**：切换到工作模式后，出现任务面板，任务可分配给指定成员，成员执行后状态更新为"完成"
- [ ] `[AUTO]` **添加专家成员**：从162专家库选择一位并添加，该成员出现在成员列表，发送 `@该成员` 能收到回复

### 3. 发送框

- [ ] `[MANUAL]` **自适应高度**：输入内容超过1行自动扩展，超过15行出现内部滚动条，输入框外部高度不再增加
- [ ] `[AUTO]` **Enter/Shift+Enter**：按 Enter 触发发送事件（消息条数+1），按 Shift+Enter 插入换行符（消息条数不变）
- [ ] `[MANUAL]` **拖拽附件**：将文件拖入输入框，文件名出现在输入框上方附件列表，大小超过 20MB 显示错误提示"文件过大"
- [ ] `[MANUAL]` **润色功能**：输入"帮我写个代码"后点击润色，输入框内容被替换为优化后的 prompt，不自动发送，用户仍可修改
- [ ] `[AUTO]` **语音输入**：按住语音按钮说话，松开后输入框出现转写文字，延迟 < 5 秒（本地 Whisper.cpp）
