# OpenAGI-MVP核心骨架树-思维导图-验收标准

## 1. 文档目的

本文档用于将《OpenAGI MVP 核心骨架树—思维导图》转化为可执行、可演示、可判定的验收标准，作为 OpenAGI MVP 核心骨架版本的评审基线。

本标准关注的是“是否达到可验收的产品骨架与最小可用闭环”，而不是“是否已经做到最终完整版”。因此，本文同时定义两条口径：

- **骨架可用级**：可作为当前 MVP 上线/内测的通过线。
- **全量兑现级**：与思维导图中的数量、覆盖度、生态规模完全对齐的目标线。

凡脑图中明确标注 **Phase 2 / Phase 4 / 未来 / 规模化** 的内容，不作为当前 MVP 的阻塞验收项，但必须在架构、接口或配置层预留扩展位。

---

## 2. 验收基线

### 2.1 来源基线

本标准以《OpenAGI MVP 核心骨架树—思维导图》为唯一范围基线，覆盖以下一级域：

1. Layer 1: LLM 大脑层
2. Layer 2: Agent 核心引擎
3. Layer 3: 永远记忆系统
4. Layer 4: 永生与分布式层
5. Layer 5: 工具与扩展层
6. 巡检 AI（Commander Core）
7. 人格系统（Persona Engine）
8. 聊天系统
9. 技能系统（Skill Engine）
10. 免 API 浏览器搜索（Browser Engine）
11. 宪法与权限系统
12. 数字伴侣系统（Persona Embodiment）
13. 用户触达层
14. 首页
15. 技术栈
16. 参考应用

### 2.2 验收对象

验收对象不仅包括界面，还包括：

- 配置项是否存在且可修改
- 核心流程是否能跑通
- 失败/回退/熔断是否生效
- 数据是否持久化且可恢复
- 权限与审计是否有效
- 日志、状态、提示是否可观测
- 扩展点是否具备继续演进能力

## 3. 验收分级与通过规则

### 3.1 分级定义

#### A. 骨架可用级（当前 MVP 通过线）
满足以下四类要求：

1. **结构存在**：模块、入口、配置、状态、数据结构齐全。
2. **主链路可用**：至少 1 条真实用户路径可闭环跑通。
3. **异常可控**：核心失败场景存在重试、回退、提示或阻断。
4. **可扩展**：后续数量扩张、生态接入、分布式演进不需要推翻现有架构。

#### B. 全量兑现级（完整目标线）
在骨架可用级通过的基础上，进一步满足思维导图中出现的数量、覆盖面、生态对接、平台数量、人格数量、工具数量等明确目标。

### 3.2 阻塞规则

以下任一情况出现，则判定为**当前版本不通过**：

1. 任一 P0 主链路无法完成。
2. L2/L3 操作未按权限规则执行确认、审计或阻断。
3. L4 禁止项可被绕过执行。
4. 主模型故障时没有自动回退，且用户无明确可见提示。
5. 会话、记忆、核心状态在重启后出现不可恢复的数据丢失。
6. 敏感信息在界面、日志、异常栈、导出文件中明文泄露。

### 3.3 建议优先级

- **P0**：不通过即不可上线。
- **P1**：建议本期完成；允许存在少量缺陷，但不可破坏主链路。
- **P2**：体验增强项，可在后续版本补齐。
- **F**：未来项/后续阶段项，仅要求有设计预留，不阻塞 MVP。

## 4. 范围边界

### 4.1 当前必须纳入验收的内容

1. 单机可部署、可恢复的 OpenAGI MVP 核心骨架。
2. 首页、聊天、模型路由、多核治理、记忆、权限、工具、巡检、技能、浏览器搜索的核心闭环。
3. 至少一条“用户发起任务 → AI 执行/审计 → 写入记忆 → 可召回 → 可追踪”的完整链路。
4. 至少一条“真实外部读操作”与一条“受限真实操作”的权限拦截链路。

### 4.2 当前不作为阻塞项的内容

1. Redis/rsync 跨节点同步。
2. Docker Swarm / K8s 多节点编排。
3. PostgreSQL + pgvector 的规模化替换。
4. 未来插件市场的全面铺开。
5. 15+ 消息平台的全量接入。
6. 36 平台 103 命令的全量打通。
7. 162 位专家人格的最终内容质量打磨。
8. Live2D / VRM / 本地 Stable Diffusion 等高成本数字形象能力的全量实现。

> 说明：上述项目不阻塞 MVP，但必须有明确的数据结构、接口层或配置层预留；不能写死成一次性实现。

## 6. 通用验收要求

### 6.1 可部署性

1. 必须能以 MVP 官方推荐方式一键启动。
2. 启动完成后，至少提供：
   - Web 入口
   - 健康检查接口
   - 配置文件或环境变量模板
3. 首次启动失败时，应给出可定位的错误信息，而不是静默退出。

### 6.2 可观测性

1. 所有核心模块至少有以下可观测信息：
   - 当前状态
   - 最近一次执行结果
   - 最近一次失败原因
2. 主链路必须可通过日志或界面追踪到：
   - 触发来源
   - 使用的模型/人格/核心数
   - 工具调用
   - 权限判定
   - 回退/重试情况

### 6.3 安全与权限

1. 所有外部真实操作都必须经过权限等级判定。
2. L3 及以上必须有人机确认，且确认记录可追踪。
3. L4 项必须被系统级阻断，不允许以提示替代阻断。
4. API Key、Token、私钥等敏感信息必须遮罩显示，日志中不得明文落盘。

### 6.4 稳定性与恢复

1. 服务异常退出后应能自动拉起。
2. 自动拉起后应恢复：
   - 最近会话索引
   - 已持久化记忆
   - 核心 DNA / 心绪状态
   - 未完成任务的最小必要状态
3. 崩溃恢复后不得产生明显的状态错乱或重复执行高风险动作。

### 6.5 中文可用性

1. 核心功能需支持中文界面与中文输入。
2. 记忆检索中的中文全文检索能力必须可用。
3. 专家人格、技能市场等中文定位模块必须具备中文可读性。

## 7. 分模块验收标准

### 7.1 首页

#### P0 骨架可用级
1. 顶部导航栏存在且可切换到：深度聊天、AI 团队群聊、记忆宫殿、设置。
2. 左侧边栏支持：新建会话、历史搜索、分组展示、重命名、删除。
3. 中间主区域在新会话时显示欢迎屏、快捷提示卡片、当前状态面板。
4. 右侧面板可展开/收起，并根据上下文在“多核治理面板”和“团队成员面板”之间切换。
5. 底部状态栏至少显示：连接状态、当前模型、今日 Token 用量、巡检 AI 下次触发时间。

#### P1 全量兑现级
1. 当前模型、心绪状态、巡检 AI 状态、用户菜单完整呈现。
2. 欢迎语可随时间段变化。
3. 右键菜单、置顶、归档、版本号、快捷键提示等细节齐备。

#### 不通过条件
- 首页仅为静态展示，无法进入聊天主流程。
- 会话列表变更后不同步或刷新即丢失。

---

### 7.2 Layer 1：LLM 大脑层

#### P0 骨架可用级
1. 系统支持主模型与至少 2 个回退模型的配置与切换。
2. 支持新增中转模型配置，至少包含：API Base URL、API Key、连接测试。
3. 测试结果可展示模型名、可用状态、连接延迟。
4. 模型列表支持“设为主模型”“设为回退模型”“删除”“批量测试”。
5. 当主模型不可用时，系统必须自动切换至回退模型，并给出明确用户提示。
6. 回退逻辑必须包含指数退避，起始时间不高于 2 秒。
7. 情绪感知 Prompt 至少支持 calm / focused / anxious / crisis 四态。
8. Token 预算管理存在可配置策略，危机状态下可触发压缩策略。

#### P1 全量兑现级
1. 模型提供商覆盖脑图列出的主要来源：Claude、GPT、Grok、本地 Ollama、DeepSeek 及中转站。
2. 中转站支持一键发现可用模型。
3. 连续 3 次失败的模型自动降级到候选队尾。
4. 延迟评级、主回退排序、实时测试更新均可视化。

#### 验收方法
- 人工断开主模型配置或模拟 5xx/超时，观察是否自动切换。
- 连续触发失败 3 次，验证排序是否变化。
- 在四种心绪状态下发起同一任务，验证 Prompt 模板差异存在且方向正确。

---

### 7.3 Layer 2：Agent 核心引擎

#### P0 骨架可用级
1. 支持 1-4 核治理模式切换。
2. 至少可配置 Core-0 执行者与 1 个审计者；每个核心具备模型、人格、温度等基础参数。
3. 权限升级规则生效：L1 至少升为 2 核，L2 至少升为 3 核，L3 强制 4 核并要求人机双签，L4 永久禁止。
4. 提供省钱/平衡/安全等预设方案，且切换后参数可见。
5. ReAct+Plan 执行循环在日志或界面中可追踪。
6. HeartEngine 至少具备 entropy、valence 两轴与四级状态转换。
7. 上下文压缩功能存在，长对话触发后能输出压缩摘要，并保留决策、代码、错误信息。

#### P1 全量兑现级
1. 4 个核心均可单独配置。
2. 17 类事件权重可配置且可驱动心绪变化。
3. 自主进化循环至少能产出“学习结果 / DNA 更新建议 / 技能创建建议”之一。
4. 长对话压缩率达到 60%~80%，且抽样关键信息保留准确率不低于 90%。

#### 不通过条件
- 核数切换只改变 UI，不影响实际执行治理链路。
- L3 操作未被强制升级为 4 核并要求确认。
- 压缩后丢失关键决策，导致后续执行偏离。

---

### 7.4 Layer 3：永远记忆系统

#### P0 骨架可用级
1. 具备 L0 热记忆、L1 温记忆、L2 冷记忆、L3 核心记忆四层结构。
2. 会话结束后，L0 内容可转入后续层，不得全部丢失。
3. L1 至少具备基于语义相似度的召回能力。
4. L2 至少具备历史归档能力，且可被显式查询。
5. L3 核心记忆可持久化，重启后仍会被注入系统上下文。
6. 支持混合检索，至少包含：语义检索、全文检索、去重机制。
7. 支持梦境蒸馏的手动触发；蒸馏结果可写入日志、记忆或日记。

#### P1 全量兑现级
1. L1 使用相似度阈值唤醒；默认阈值与脑图一致或可配置。
2. L1/L2 的衰退、提升、归档机制完整可见。
3. Light Sleep / REM Sleep / Deep Dreaming 三阶段调度周期分别为 2 小时、6 小时、24 小时。
4. 梦境日记每日生成，至少包含：关键发现、连接、矛盾、能力更新。
5. 混合检索支持 BM25 中文分词、MMR 去重、时间衰退加权。

#### 验收方法
- 结束一段对话后，重新查询相关主题，验证记忆可被召回。
- 注入新旧冲突事实，执行 REM/Deep Dreaming，验证是否能发现矛盾。
- 重启服务，验证核心记忆和历史记忆仍可查询。

---

### 7.5 Layer 4：永生与分布式层

#### P0 骨架可用级
1. 存在心跳调度器，并可根据心绪状态调整心跳频率。
2. 调度器至少承担：状态快照、蒸馏触发、健康检查、缓存清理中的 3 项。
3. 本地持久化至少包括：SQLite 数据、JSON 状态、未提交事务日志或等价恢复机制。
4. 服务异常中断后，可自动恢复并继续使用。
5. Docker Compose 单机部署可用。

#### F 后续项
1. Redis 实时状态共享。
2. rsync 冷记忆同步。
3. Docker Swarm / K8s 多节点。

#### 不通过条件
- 服务恢复后会话索引丢失。
- 心跳调度器只存在配置，不实际执行任何任务。

---

### 7.6 Layer 5：工具与扩展层

#### P0 骨架可用级
1. 动态工具注册表可用，至少支持注册、注销、按权限列出工具。
2. 工具层至少覆盖以下六类中的每一类 1 个真实工具：
   - 文件操作
   - 执行
   - 网络
   - 代理
   - 交互
   - MCP / Git
3. 工具调用前后可挂接 Hook。
4. 至少存在 1 条文件类 Hook、1 条 API 类 Hook、1 条 Agent 类 Hook 的真实触发记录。
5. MCP 存在统一接入点，能识别外部工具资源。

#### P1 全量兑现级
1. 脑图列出的 23 种标准工具全部可识别、可调用、可追踪。
2. Hook 覆盖文件、Git、Agent、API 四类。
3. 插件管理器支持安装、启用、禁用、卸载。
4. 第三方工具可通过 MCP 以统一协议接入。

#### 不通过条件
- 工具权限标记不存在或不生效。
- Hook 只在文档中存在，实际不触发。

---

### 7.7 巡检 AI（Commander Core）

#### P0 骨架可用级
1. 支持定时触发与事件触发两种模式。
2. 默认发送模式为草稿模式。
3. 在草稿模式下，巡检结果必须写入发送框或等价待确认区域，不可直接执行。
4. 自动执行模式下必须受权限矩阵约束。
5. 当任务执行中时，巡检可延迟；当任务完成或危机事件出现时，可立即触发。
6. 巡检内容至少支持：项目进展总结、阻塞识别、未完成任务检查、下一步规划。

#### P1 全量兑现级
1. 预设周期、自定义周期、事件触发器、智能等待逻辑完整。
2. 外部状态检查与过期记忆清理可配置。
3. 巡检历史可查询。

#### 不通过条件
- 草稿模式仍绕过用户确认直接发送。
- 巡检在高风险场景下无视权限等级自动执行。

---

### 7.8 人格系统（Persona Engine）

#### P0 骨架可用级
1. 支持三种设定方式：预设人格、专家模板、自定义人格。
2. 人格至少具备名称、描述/Prompt、温度等可配置项。
3. 支持搜索、预览、选择、复制到当前核心。
4. 中文显示可用。

#### P1 全量兑现级
1. 13 个专家域分类齐全。
2. 全中文 162 位专家 AI 人格可选。
3. 支持从现有核心一键复制人格配置。

#### 不通过条件
- 人格切换后对输出风格或策略无实际影响。
- 自定义人格无法保存或无法再次选用。

---

### 7.9 聊天系统

#### P0 骨架可用级
1. 支持“深度聊天”和“AI 团队群聊”两种模式切换。
2. 深度聊天可与多核治理引擎联动。
3. 群聊模式支持 @ 指定成员回复。
4. 消息区支持 Markdown 渲染、流式输出、代码高亮、复制按钮。
5. 输入区支持多行输入、附件上传、图片上传、发送与停止。
6. AI 工作中状态可见，至少包含耗时提示与“停止”操作。
7. 群聊成员可添加、删除、配置模型/人格/技能。

#### P1 全量兑现级
1. 支持 @ 全体、AI 相互 @、轮次上限、旁听者积累上下文。
2. 讨论模式与工作模式可切换。
3. 预设团队模板不少于 6 个。
4. 支持引用回复、Token 用量标签、截图/粘贴图片。
5. DAG 依赖调度可驱动团队并行执行。

#### 不通过条件
- 切换为群聊模式后仍是单体回复，不存在成员视角。
- “停止”按钮仅改 UI，不中断实际生成或执行。

---

### 7.10 技能系统（Skill Engine）

#### P0 骨架可用级
1. 技能具备统一定义结构：名称、描述、输入输出、执行逻辑、权限级别、标签。
2. 至少存在内置技能集，可被启用并执行。
3. 存在技能市场入口，且至少能浏览、搜索、安装技能。
4. 自动下载逻辑受权限等级约束：L0/L1 可自动，L2+ 需确认。
5. 重复操作识别机制存在，能够生成“技能候选”或“技能草稿”。

#### P1 全量兑现级
1. 官方中文镜像市场与 CocoLoop 市场可切换。
2. 兼容 OpenClaw / agentskills.io 等标准格式。
3. 重复操作达到阈值后，可自动抽取为技能并经过审查启用。
4. 使用后的反馈可反向优化技能。

#### 不通过条件
- 技能只是提示词收藏，无法作为可执行能力调用。
- 安装高权限技能时不经过确认。

---

### 7.11 免 API 浏览器搜索（Browser Engine）

#### P0 骨架可用级
1. 具备 AI 层 → 本地守护进程 → 真实浏览器的三层架构或等价抽象。
2. 支持至少 1 条真实网页搜索链路。
3. 平台能力以“平台 + 命令/动作”形式建模，不允许写死为单一站点脚本。
4. 权限等级至少覆盖只读搜索、读取登录态数据、表单点击三类。
5. 搜索结果可写入温记忆或等价近期记忆层。
6. 对同一问题具备重复搜索抑制机制。

#### P1 全量兑现级
1. 六大平台类别齐全：搜索引擎、社交媒体、开发平台、视频平台、财经、学术。
2. 36 平台 103 命令达到脑图目标。
3. L3 支付操作要求人机双签，L4 银行密码页面永久禁止。

#### 不通过条件
- 搜索能力依赖纯手写 mock，无法访问真实页面。
- 高风险浏览器动作没有权限升级。

---

### 7.12 宪法与权限系统

#### P0 骨架可用级
1. 存在 L0-L4 权限熔断矩阵，并在执行前强制判定。
2. L0 自动执行、L1 受限外部读、L2 双 AI 审、L3 人机双签、L4 永久禁止的规则可被验证。
3. 宪法规则至少包含：目标定义、约束条件、价值权重。
4. 敏感信息检测存在，检测到密钥、私钥等信息时必须升级或阻断。
5. 含密钥提交、真实危险删除、真实高风险转账等动作不可绕过规则。

#### P1 全量兑现级
1. 价值权重可配置并影响决策。
2. 审计记录可查询、可追溯。
3. 常见敏感模式库齐全，并支持扩展。

#### 不通过条件
- L4 规则被前端参数绕过。
- 检测到敏感信息后仅做提示，不升级权限或阻断。

---

### 7.13 数字伴侣系统（Persona Embodiment）

#### P1 骨架可用级
1. 至少支持一种视觉形象承载方式（静态头像即可）。
2. 至少支持一种语音输入或语音输出能力。
3. 情绪状态与表达方式存在映射关系，哪怕是最小版本。
4. 亲密沟通模式可切换，至少具备专业助手与朋友两档。

#### P2 / 全量兑现级
1. 五感系统完整：眼睛、耳朵、嘴巴、心、手均有落地点。
2. 支持 Live2D / VRM / 本地 SD 自拍 / 本地 STT/TTS / 声音克隆。
3. AI 自拍支持自然语言触发与多模式发送。

#### 不通过条件
- 宣称存在数字伴侣模式，但仅是更换头像，没有任何交互层差异。

### 7.14 用户触达层

#### P0 骨架可用级
1. Web API 至少提供：主业务接口、健康检查、实时通信接口。
2. WebSocket 聊天链路可用。
3. 至少 1 个外部渠道可用，优先 Telegram。
4. 消息网关具备统一抽象，不因新增平台而改动核心业务逻辑。
5. 迁移工具命令入口存在，至少可展示帮助信息或最小迁移能力。

#### P1 全量兑现级
1. Telegram Bot 的核心命令可用。
2. 消息网关支持更多平台扩展。
3. 迁移工具支持从脑图列出的来源迁移。

#### 不通过条件
- API 可启动但核心接口不可达。
- 外部渠道接入需要改动核心聊天逻辑，说明网关抽象失败。

### 7.15 技术栈

## 8. 端到端场景验收

以下场景均为 P0，必须全部通过。

### 场景 1：首次进入并完成一次深度聊天
1. 用户进入首页，新建会话。
2. 选择当前模型，输入问题并发送。
3. AI 正常流式回复。
4. 消息进入历史会话。
5. 状态栏更新 Token 用量或等价消耗信息。

### 场景 2：主模型故障自动回退
1. 主模型故意制造失败。
2. 系统自动切换到回退模型继续完成回复。
3. 用户可见故障转移提示。
4. 日志可见失败原因与回退目标。

### 场景 3：高权限操作被升级治理并要求确认
1. 用户发起一个 L2 或 L3 操作。
2. 系统自动提升核心数。
3. L3 操作必须进入人机确认。
4. 未确认前不得执行。

### 场景 4：记忆写入与召回
1. 用户在会话中提供一个特定偏好或项目背景。
2. 结束会话或触发持久化。
3. 新会话中以相近问题提问。
4. 系统可召回相关信息并在答案中体现。

### 场景 5：工具调用与 Hook 审计
1. 用户发起需要工具的任务。
2. 工具被正确选择并执行。
3. 对应 Hook 被触发。
4. 日志可追踪完整链路。

### 场景 6：巡检 AI 产出下一步建议
1. 开启巡检 AI。
2. 在存在未完成任务时触发巡检。
3. 巡检结果输出为草稿。
4. 用户确认前不自动发送。

### 场景 7：浏览器搜索与记忆联动
1. 用户请求检索网页资料。
2. 浏览器搜索链路完成。
3. 检索结果进入近期记忆。
4. 相同问题再次触发时，系统可优先复用已有结果或降低重复搜索。

### 场景 8：异常重启后状态恢复
1. 在存在会话与记忆数据时中断服务。
2. 服务自动恢复。
3. 历史会话、核心记忆、模型配置仍然存在。
4. 不产生重复的高风险操作执行。

## 9. 非功能验收标准

| 维度 | P0 验收标准 |
|---|---|
| 性能 | 首页可进入；聊天请求有明确加载/工作状态；流式回复期间用户可停止 |
| 可靠性 | 主链路连续执行 20 次，不得出现高频无提示失败 |
| 恢复性 | 非正常中断后，可恢复到最近一次已提交状态 |
| 安全性 | 敏感信息不明文展示；L3/L4 不可绕过 |
| 可观测性 | 模型、核心、工具、权限、记忆、巡检均有可追踪记录 |
| 可扩展性 | 新模型、新工具、新平台、新人格不要求改写主干架构 |
| 中文能力 | 中文界面、中文搜索、中文记忆召回可用 |
